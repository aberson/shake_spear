# shake_spear ŌĆö Project Plan

A markdown-first creative writing workshop with shared AI prompt skills, per-story
subprojects, and a small stdlib-only CLI (`ss`). This plan was produced by `/plan-init`
from the seed spec at [`docs/seed.md`](docs/seed.md) plus a planning conversation on
2026-07-02. Where this plan and the seed conflict, **this plan wins** (the deltas are
listed in ┬¦8 Key Design Decisions).

## 1. What This Is

shake_spear is a personal creative-writing practice system: one workshop root holding
shared prompt skills, markdown templates, and a lightweight Python CLI, plus many
independent story subprojects under `projects/` that each work standalone when opened
directly in VS Code (`code .`). The AI (Claude or any assistant) acts as writing coach,
scene partner, continuity assistant, and editor ŌĆö guided by the skill files ŌĆö and does
not write the stories by default. The code never calls an AI API; all "AI" behavior is
prompt files the operator invokes. v1 is local-only: no web app, no GUI, no database,
no paid-API dependency. Optimize for momentum, creative safety, and easy stop/resume ŌĆö
not maximum output.

## 2. Stack

| Layer | Tool | Why |
|---|---|---|
| Language | Python 3.11+ | Seed requirement; durable, boring |
| Runtime deps | Standard library only (`argparse`, `pathlib`, `shutil`, `datetime`, `re`) | Seed principle: simple, durable; no YAML dep ŌĆö hand-rolled frontmatter parser |
| Env/packaging | uv (`uv sync`, `uv run ss`) with a standard `pyproject.toml` | Workspace norm; `pip install -e .` still works on other machines |
| CLI entry point | `[project.scripts] ss = "shake_spear.cli:main"` | Single short verb; `spear` alias is a one-line addition later if WSL use materializes |
| Tests | pytest (dev dep), `tmp_path`-based | Filesystem-heavy code; no mocks needed |
| Lint/type | ruff + mypy (dev deps) | Workspace quality-gate norm |
| Writing assets | Markdown files with lightweight frontmatter | The entire product surface is markdown |
| AI integration | None at runtime. `skills/*.md` prompt files + thin Claude Code wrappers under `.claude/skills/` | Tool-agnostic source of truth, slash-invocable in Claude Code |

No backend, no API routes ŌĆö the API Route Contract section is intentionally omitted.
No ports are used by this project.

## 3. Data Store

Everything is flat markdown under the workshop root. No database, no index service ŌĆö
`index.md` files are derived artifacts regenerated on demand.

### 3.1 Story project anatomy

Created by `ss new-story`. Division of labor (one source of truth, no drift surface):
`projects/_template/` holds ONLY the empty subfolder skeleton (`.gitkeep` files + the
story `.gitignore`); every story FILE is rendered fresh from `templates/` at
new-story time ŌĆö `_template` never duplicates template content.

```text
projects/<story_slug>/
  README.md            # what this story is, how to work in this folder
  AGENTS.md            # local guide for non-Claude assistants
  CLAUDE.md            # local guide Claude Code auto-loads on `code .`
  story_bible.md       # premise, tone, characters, themes (frontmatter: type/title/genre/audience/status/mode)
  active_state.md      # THE "start here" file; contains the ss:recap marker block
  continuity.md        # running continuity facts, promises, open questions
  decisions.md         # decision log (what was decided and why)
  index.md             # DERIVED ŌĆö regenerated by `ss index`, safe to overwrite
  .claude/skills/      # GENERATED story-local wrappers so slash skills work when this folder is opened directly
  .gitignore           # junk only (never ignores drafts)
  characters/          # one file per character (character_profile template)
  world/               # one file per world element (world_element template)
  scenes/              # one scene card per scene (scene_card template)
  drafts/              # prose drafts; NEVER auto-modified by tooling
  sessions/            # dated session logs (session_log template)
  feedback/            # feedback notes
  revisions/           # revision plans
  prompts/             # story-specific prompt files (operator-authored; seed ┬¦15)
  exports/             # DERIVED ŌĆö `ss export` writes manuscript.md here
```

### 3.2 Identifiers and file naming

| Entity | ID / filename format | Example |
|---|---|---|
| Story project | slug: `[a-z0-9_]+`, from `--slug` or slugified title | `kids_space_bakery` |
| Scene / character / world element | `<slugified_title>.md` in the entity folder | `scenes/rocket_finds_the_moon_oven.md` |
| Session log | `YYYY-MM-DD_<type>_<minutes>min.md` (date = local date) | `sessions/2026-07-02_scene_45min.md` |
| Session-log collision | `_b`, `_c`, ŌĆ” `_z` appended before `.md` when the target exists (session logs ONLY ŌĆö named entities refuse instead, see ┬¦3.4) | `2026-07-02_scene_45min_b.md` |
| Backup (forced overwrite) | `<original>.bak-YYYYMMDDHHMMSS` | `story_bible.md.bak-20260702153000` |

**Slugify algorithm** (one implementation in `utils.py`, used everywhere): lowercase ŌåÆ
every run of non-`[a-z0-9]` characters becomes a single `_` ŌåÆ strip leading/trailing
`_`. Empty result is a usage error (exit 1).

### 3.3 Frontmatter grammar (hand-rolled, no YAML dependency)

A file *may* begin with a frontmatter block: first line exactly `---`, block ends at
the next line exactly `---`. Inside: `key: value` pairs. A value of the form
`[a, b, c]` parses to a list of stripped strings; everything else is a plain string
(may be empty). Unparseable lines are ignored. Files without frontmatter are legal
everywhere; the indexer falls back to filename and first non-empty line.

```markdown
---
type: scene
title: Rocket finds the moon oven
status: planned
characters: [Nova, Rocket]
tags: []
---
```

### 3.4 Overwrite, backup, and creative-safety policy

- `utils.safe_write()` takes an explicit `mode` argument ŌĆö the two strategies are
  never conflated; every writer declares one:
  - `mode="refuse"` ŌĆö named-entity writers (`new-story` target dir, `scene`,
    `character`, `world`): if the target exists, exit 2 with a clear message unless
    `--force` is passed; `--force` **still** writes a timestamped `.bak-` copy next
    to the target first, then overwrites.
  - `mode="suffix"` ŌĆö dated session logs (`session`, `daily`): if the target exists,
    auto-append `_b`ŌĆ”`_z` and create a new file; never refuses, never takes
    `--force`.
- Exceptions ŌĆö files that are *derived artifacts*, regenerated freely without
  `--force`: `index.md`, `exports/manuscript.md`.
- `ss recap` never rewrites `active_state.md` wholesale: it only replaces the text
  between `<!-- ss:recap:start -->` and `<!-- ss:recap:end -->` markers (appending
  the marker block at the end of the file if absent). All operator-written content
  is preserved byte-for-byte.
- `drafts/` is never written or modified by any command except `ss export` *reading*
  it. Never delete creative writing automatically.
- All file I/O uses explicit `encoding="utf-8"` and `newline="\n"` (Windows-safe;
  see workspace shell rules on BOM/CRLF landmines).

### 3.5 Git model (public tools, private stories)

- The workshop root is **one public GitHub repo** (published via `/repo-init`).
- Root `.gitignore` ignores `projects/*` with re-includes `!projects/_template/`
  and `!projects/example_kids_story/` ŌĆö the template and the committed sample are
  the only stories in the public repo.
- Every real story is its **own nested git repo**: `ss new-story` runs `git init`
  in the new folder + one initial commit (`--no-git` to skip; skipped gracefully
  with a notice if git isn't on PATH). Stories are pushed to GitHub as private
  repos per-story when the operator wants backup, and any single story can be
  flipped public later without touching the others.
- The parent `dev/` coding-root repo leaves `shake_spear/` untracked (same as
  other nested repos like `brickomancer/`).

## 4. CLI Command Contract

Global conventions:

- Invocation: `uv run ss <command> ŌĆ”` (or plain `ss` in an activated venv).
- Every command that targets a story accepts the project as `projects/<slug>`, a bare
  slug, or an absolute path ŌĆö and the argument is **optional when cwd is inside a
  story project** (detection: walk up from cwd looking for a directory containing
  both `story_bible.md` and `active_state.md`).
- Exit codes: `0` success ┬Ę `1` usage/validation error (bad slug, missing project,
  unknown type) ┬Ę `2` refused overwrite (target exists, no `--force`).
- Every command prints the path(s) it created or updated, one per line.

| Command | Signature | Behavior |
|---|---|---|
| `ss new-story` | `ss new-story "Title" [--slug S] [--genre G] [--mode M] [--force] [--no-git]` | Scaffold `projects/<slug>/` from `projects/_template/`: all subfolders, local README/AGENTS.md/CLAUDE.md (rendered from `templates/local_*`), `story_bible.md` (title/genre/mode filled into frontmatter), `active_state.md`, `continuity.md`, `decisions.md`, `index.md`, story-local `.claude/skills/` wrappers, story `.gitignore`; then `git init` + initial commit |
| `ss session` | `ss session [PROJECT] [--type T] [--minutes N]` | Create `sessions/<date>_<type>_<minutes>min.md` from `templates/session_log.md`, frontmatter filled, with a short summary pulled from `active_state.md` (Current status + Next tiny action sections; if the file or either section is missing/empty, the summary reads "(no active state yet)" ŌĆö never an error) and relative links to the type-relevant shared skills (mapping: Appendix F). Same-day collision ŌåÆ `_b` suffix (`mode="suffix"`, no `--force`). Defaults: `--type scene`, `--minutes 45`. Types: `scene, revision, character, worldbuilding, dialogue, kids, mystery, freewrite` (free-form accepted, slugified into the filename) |
| `ss scene` | `ss scene [PROJECT] "Title" [--force]` | Create `scenes/<slug>.md` from `templates/scene_card.md` with title in frontmatter (`mode="refuse"`: exists ŌåÆ exit 2 unless `--force`) |
| `ss character` | `ss character [PROJECT] "Name" [--force]` | Create `characters/<slug>.md` from `templates/character_profile.md` with name in frontmatter (`mode="refuse"`) |
| `ss world` | `ss world [PROJECT] "Name" [--force]` | Create `world/<slug>.md` from `templates/world_element.md` with name in frontmatter (`mode="refuse"`) |
| `ss index` | `ss index [PROJECT]` | Regenerate `index.md` (derived): sections for story files, characters, world elements, scenes, drafts, sessions, feedback, revisions, recently-modified (top 10 by mtime); per file: title (frontmatter or filename), type, status, relative path, first non-empty body line. Scan excludes dot-directories (`.claude/`, `.git/`) |
| `ss recap` | `ss recap [PROJECT]` | Read `active_state.md`, `continuity.md`, the latest 3 session logs, and index data; rewrite ONLY the `ss:recap` marker block in `active_state.md` with a "Start here next time" summary (current state, what changed, open loops, best next scene, one tiny next action). With zero sessions / empty inputs it still writes the block with "no sessions yet" placeholders and exits 0 |
| `ss status` | `ss status [PROJECT]` | Print (stdout only, writes nothing): Current status, Current scene/chapter, Open loops, Next tiny action from `active_state.md` + newest session filename |
| `ss list-projects` | `ss list-projects` | Table of all `projects/*` (excluding `_template`): slug, title, genre, status from `story_bible.md` frontmatter |
| `ss daily` | `ss daily [PROJECT]` | Sugar for `ss session PROJECT --type freewrite --minutes 15` |
| `ss export` | `ss export [PROJECT] [--out PATH]` | Concatenate `drafts/*.md` in lexicographic filename order into `exports/manuscript.md` (derived; each part preceded by an `## <filename>` heading). Read-only w.r.t. drafts |

## 8. Key Design Decisions

- **Public tools, private stories, per-story flip.** One public workshop repo;
  `projects/*` gitignored; each story is its own nested git repo initialized by
  `ss new-story`, pushed private per-story on demand, individually flippable to
  public later. Chosen over a single private stories repo (extraction pain) and
  over no-git stories (no backup). Operator decision 2026-07-02.
- **CLI named `ss`, single entry point.**
  Known, accepted: `ss` collides with the Linux/WSL iproute2 socket-statistics tool ŌĆö
  irrelevant for this Windows-first project and `uv run ss` never collides; a
  `spear` alias is a one-line `[project.scripts]` addition if ever needed.
- **Dual skill wiring.** `skills/*.md` are the tool-agnostic source of truth
  (usable from any assistant); `.claude/skills/` wrappers add slash-invocation.
  Wrappers contain no content of their own ŌĆö no drift surface. Story-local wrappers
  are *generated* (copied logic, no symlinks) so `code .` in a story folder keeps
  slash skills working.
- **`CLAUDE.md`, not `Claude.md`** (root and story-local): Claude Code auto-loads
  the uppercase name; `AGENTS.md` covers other tools. Delta from seed ┬¦4/┬¦6/┬¦7.
- **uv-managed, stdlib runtime** (seed said venv+pip): workspace norm; pyproject
  stays standard so `pip install -e .` also works. Runtime deps: none. Dev deps:
  pytest, ruff, mypy.
- **Creative safety as a hard invariant, not documentation**: refuse-overwrite +
  `--force`-with-backup + marker-block-only recap + never-touch-drafts are enforced
  in `utils.safe_write()`/`session.recap()` (one choke point each) and covered by
  tests ŌĆö per the workspace rule that docs are not a control.
- **Derived vs authored files** are explicitly split: `index.md` and
  `exports/manuscript.md` are the only regenerate-freely files; everything else is
  operator-owned.
- **Hand-rolled frontmatter** (seed requirement): the grammar in ┬¦3.3 is the whole
  contract ŌĆö one parser in `utils.py`, used by indexer, creators, and recap
  (one source of truth for the shape; no duplicate parsing).
- **All v1 extras in scope** (operator decision): `list-projects`, `status`,
  `daily`, `export`, VS Code tasks, committed sample story, per-story `prompts/`
  folder, tag conventions (documented in the story README template, not a separate
  `tags.md`). Sample = ONE committed kids story; the seed's second (mystery) sample
  is deferred ŌĆö M1 creates a *real* mystery story instead (Appendix E).

### A. Session filename grammar

`<YYYY-MM-DD>_<type_slug>_<minutes>min.md` ŌĆö date is the local date at creation;
`type_slug` is the slugified `--type` value; collision handling appends `_b`ŌĆ”`_z`.

### B. `ss:recap` marker block (shipped inside `templates/active_state.md`)

```markdown
<!-- ss:recap:start -->
_(ss recap writes the "start here next time" summary here; everything outside

<!-- ss:recap:end -->
```

### C. Story-local wrapper example (generated by `ss new-story`)

`projects/<slug>/.claude/skills/immersive-session/SKILL.md`:

```markdown
---
name: immersive-session
description: Guide a focused 45-minute writing session for this story.
user-invocable: true
---

This folder is one story inside the shake_spear workshop. Read
`../../skills/immersive_session.md` (relative to this story's root) and follow
it. First read `active_state.md`, `story_bible.md`, and `continuity.md` here.
If the shared skills path does not exist (standalone clone), say so and coach
from the local story files alone.
```

### D. Frontmatter parser contract (normative)

- Block delimiters: first file line exactly `---`; ends at next line exactly `---`.
- Pairs: `key: value` ŌĆö key `[A-Za-z_][A-Za-z0-9_-]*` (hyphens allowed: the Claude
  Code wrapper key `user-invocable` must survive parse/render round-trips), value =
  rest of line stripped.
- Lists: value matching `[ ŌĆ” ]` splits on `,`, strips items; empty brackets ŌåÆ `[]`.
- Everything is a string; no type coercion, no nesting, no multiline values.
- Malformed lines are skipped silently; absent frontmatter ŌåÆ `{}` (callers fall back
  to filename + first non-empty body line).
- `render()` is the inverse and round-trips what `parse()` accepts.
